üìö Git Basic Concepts

Understanding the Four Areas:

1. Working Directory: Where you edit your files. These are untracked changes.

2. Staging Area: Prepared changes ready for commit. Preview of next commit.

3. Local Repository: Your computer's Git history with all commits.

4. Remote Repository: Server-hosted repository (GitHub) for collaboration.

git add filename.txt # Move to staging
git commit -m "message" # Save to local repo
git push origin main # Upload to remote
git pull origin main # Download from remote

üîÑ Complete Git Workflow

Step-by-Step Process:

1. Initialize: Create new Git repository with git init

2. Edit: Make changes to your files

3. Stage: Select changes to include in commit

4. Commit: Save snapshot with descriptive message

5. Push: Upload commits to remote server

6. Pull: Download latest changes from remote

git init # Initialize repository
git add . # Stage all changes
git commit -m "Initial commit" # Commit changes
git remote add origin URL # Link remote repo
git push -u origin main # Push to remote
git pull origin main # Get latest changes
üí° Key Concept: Git workflow is cyclical. After pulling, you edit, stage, commit, and push again.

üåø Git Branching Strategy

Branching Concepts:

Branch: Independent line of development. Like a parallel universe for experiments.

main/master: Default primary branch with production-ready code.

Feature Branches: Created for new features without affecting main.

Merge: Combining changes from one branch into another.

git branch # List branches
git branch feature-login # Create new branch
git checkout feature-login # Switch to branch
git checkout -b new-feature # Create and switch
git merge feature-login # Merge into current
git branch -d feature-login # Delete branch
‚ö†Ô∏è Best Practice: Always create new branch for each feature. Keep main stable!

üêô GitHub Collaboration Flow

GitHub Collaboration Steps:

1. Fork: Create your copy of repository on GitHub

2. Clone: Download your fork to local computer

3. Make Changes: Edit, commit, push to your fork

4. Pull Request: Ask owner to review and merge

5. Merge: Owner accepts and merges your changes

# Fork on GitHub (use Fork button)
git clone https://github.com/YOU/repo.git
cd repo
git checkout -b my-feature
# Make changes...
git add .
git commit -m "Add feature"
git push origin my-feature
# Create Pull Request on GitHub

üéì Beginners Guide to Git

‚úÖ DO These Things

1. Write Clear Commit Messages ‚úì "Add user login feature"
‚úì "Fix navigation bug in mobile"
‚úì "Update README with setup steps"
2. Commit Often, Push Regularly

Save your work frequently. Don't wait until everything is perfect!

3. Always Pull Before You Start git pull origin main

Get latest changes before starting work!

4. Use .gitignore File

Don't commit sensitive files, large files, or temporary files

node_modules/
.env
*.log
.DS_Store
5. Check Status Before Commit git status
git diff

Review what you're about to commit!

‚ùå DON'T Do These

1. Bad Commit Messages ‚úó "update"
‚úó "asdf"
‚úó "fixed stuff"
‚úó "final version"
2. Commit Passwords or API Keys

NEVER commit .env files, passwords, or secret keys. Use .gitignore!

3. Commit Large Binary Files

Don't commit videos, large images, or compiled files. Use Git LFS if needed.

4. Force Push to Shared Branches git push --force # ‚ö†Ô∏è DANGEROUS!

This can destroy team members' work!

5. Work Directly on Main Branch

Always create feature branches. Keep main stable!

üõ†Ô∏è Pro Commands You Should Know

Undo Mistakes

# Undo last commit (keep changes)
git reset --soft HEAD~1

# Discard local changes
git checkout -- file.txt

# Revert a commit (safe)
git revert <commit-hash>

Stash Work

# Save work temporarily
git stash

# See stashed changes
git stash list

# Apply stashed changes
git stash pop

Cherry Pick

# Apply specific commit
git cherry-pick <commit-hash>

# Useful for selective merges

Search History

# Search in commit messages
git log --grep="bug"

# Find who changed a line
git blame file.txt

‚ö†Ô∏è Common Project Mistakes

‚ùå Committing node_modules/

This folder can be HUGE (100MB+). Always add to .gitignore!

# If already committed, remove it:
git rm -r --cached node_modules
echo "node_modules/" >> .gitignore
git commit -m "Remove node_modules"

‚ùå Committing .env Files

Contains sensitive API keys and passwords!

# Create .env.example instead:
DATABASE_URL=your_database_url_here
API_KEY=your_api_key_here

# Add .env to .gitignore

‚ùå No Branch Protection

Enable branch protection on GitHub for main branch:

  • Require pull request reviews
  • Require status checks to pass
  • Prevent force pushes
  • Require signed commits
