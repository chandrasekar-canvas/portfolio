üìö Git Basic Concepts

Understanding the Four Areas:

1. Working Directory: Where you edit your files. These are untracked changes.

2. Staging Area (Index): Prepared changes ready for commit. Think of it as a "preview" of your next commit.

3. Local Repository: Your computer's complete Git history with all commits.

4. Remote Repository: Server-hosted repository (GitHub) for collaboration and backup.

git add filename.txt # Move to staging
git commit -m "message" # Save to local repo
git push origin main # Upload to remote
git pull origin main # Download from remote

üîÑ Complete Git Workflow

# Complete workflow commands
git init # Initialize repository
git add . # Stage all changes
git commit -m "Initial commit" # Commit changes
git remote add origin URL # Link remote repo
git push -u origin main # Push to remote
git pull origin main # Get latest changes
üí° Key Concept: Git workflow is cyclical. After pulling changes, you edit, stage, commit, and push again. This cycle continues throughout development.

üåø Git Branching Strategy

Branching Concepts:

Branch: An independent line of development. Think of it as a parallel universe where you can experiment.

main/master: The default primary branch containing production-ready code.

Feature Branches: Created for developing new features without affecting main code.

Merge: Combining changes from one branch into another.

# Branching commands
git branch # List branches
git branch feature-login # Create new branch
git checkout feature-login # Switch to branch
git checkout -b new-feature # Create and switch
git merge feature-login # Merge branch into current
git branch -d feature-login # Delete branch
‚ö†Ô∏è Best Practice: Always create a new branch for each feature or bug fix. Keep main branch stable. Never commit directly to main in team projects!

üêô GitHub Collaboration Flow

GitHub Collaboration Steps:

1. Fork: Create your own copy of someone's repository on GitHub.

2. Clone: Download your fork to your local computer.

3. Make Changes: Edit files, commit locally, then push to your fork.

4. Pull Request (PR): Ask the original owner to review and merge your changes.

5. Merge: After review, changes are merged into original repository.

# Fork and contribute workflow
# 1. Fork on GitHub (use button)
git clone https://github.com/YOU/repo.git
cd repo
git checkout -b my-feature
# Make changes...
git add .
git commit -m "Add new feature"
git push origin my-feature
# 2. Create Pull Request on GitHub

üë• Team Collaboration Scenarios

Common Collaboration Scenarios:

Scenario 1: Simple Push/Pull
- Alice pushes her changes to GitHub
- Bob pulls those changes before starting work
- Everyone stays synchronized

Scenario 2: Merge Conflicts
- Alice and Bob edit the same file
- Bob pushes first
- Alice must pull, resolve conflicts, then push

Scenario 3: Feature Branches
- Each developer works on separate branches
- Submit Pull Requests for review
- Merge after approval
